# 13 · 硬件端侧部署评估: Nano / Jetson / Qualcomm / YOLO 加速器

> 日期:2026-06-17
>
> 前提:假定当前 AiMe 已有基础避障能力、建图能力。本文只评估新版最低 VLFM 语义侧组件在端侧怎么落地,不把 SLAM / 避障 / 底盘控制的算力再重复计入。

---

## 13.0 结论先行

1. **老 Jetson Nano 4GB 不建议作为完整 VLFM 端侧机。** 无 RGB-D 时你的实测模型合计已经是 `4378 MiB = 4.28 GiB`,超过老 Nano 的 4GB 统一内存;即使未来有 RGB-D 降到 `2202 MiB = 2.15 GiB`,还要给系统、ROS、Python、CUDA/TensorRT context、图像缓存、地图留空间,工程上仍很紧。
2. **如果 leader 说的 "Nano" 是 Jetson Orin Nano Super 8GB,结论不同:** 未来 RGB-D 版本可以把它当最低试验板;无 RGB-D 版本不建议作为交付目标,因为 ZoeDepth 会吃掉最大头的显存和延迟预算。
3. **推荐硬件下限:** 无 RGB-D 版本按 **Jetson Orin NX 16GB** 起步;有 RGB-D 版本可以尝试 **Jetson Orin Nano Super 8GB**,更稳的是 Orin NX 8GB/16GB。要留足调试余量、相机/日志/多进程余量,上 **AGX Orin 32GB/64GB** 更舒服。
4. **高通、Hailo、Rockchip、CPU/OpenVINO 主要是 YOLO 检测服务的适配路线,不是整套 VLFM 的直接替代。** 它们能把 `/yolov7` 这个检测服务换掉,但 MobileSAM / SigLIP2 / ZoeDepth 不会自动被这些 NPU/ASIC 加速。
5. **最大硬件优化不是换 YOLO 芯片,而是上 RGB-D。** RGB-D 直接去掉 ZoeDepth,你的模型常驻合计从 `4.28 GiB` 降到 `2.15 GiB`,同时减少单目深度估计的延迟和误差源。

## 13.1 你的实测显存怎么读

### 无 RGB-D 版本

| 组件 | 峰值显存 |
|---|---:|
| MobileSAM | 1018 MiB |
| SigLIP2 ITM | 666 MiB |
| ZoeDepth | 2176 MiB |
| YOLO26L TRT | 518 MiB |
| **合计** | **4378 MiB = 4.28 GiB** |

### 未来 RGB-D 版本

| 组件 | 峰值显存 |
|---|---:|
| MobileSAM | 1018 MiB |
| SigLIP2 ITM | 666 MiB |
| YOLO26L TRT | 518 MiB |
| **合计** | **2202 MiB = 2.15 GiB** |

### 工程余量判断

| 模式 | 模型合计 | 最低可尝试 | 推荐交付 |
|---|---:|---|---|
| 无 RGB-D | 4.28 GiB | 8GB 统一内存设备,且要低帧率/分时调度 | 16GB 及以上 |
| 有 RGB-D | 2.15 GiB | 4GB 理论可装,但非常紧 | 8GB 及以上 |

注意两件事:

- Jetson 的 GPU 和 CPU 用的是**统一内存**,不是桌面显卡那种独立显存。模型吃掉的 "显存" 本质上也会挤占系统内存。
- 上面合计只算模型峰值,不包含系统进程、ROS、Python 解释器、多模型服务的 CUDA context、TensorRT workspace、相机 buffer、地图/轨迹缓存、日志和可视化。

## 13.2 Nano 到底能不能跑

这里最容易误会:市面上有两个常被口头叫 "Nano" 的东西。

| 叫法 | 核心差异 | 对本项目的判断 |
|---|---|---|
| **Jetson Nano 4GB** | 老 Maxwell GPU,4GB LPDDR4,25.6GB/s;软件栈停在 JetPack 4.x/L4T 32.x | **完整 VLFM 不建议/基本跑不起来** |
| **Jetson Orin Nano Super 8GB** | Ampere GPU + Tensor Cores,8GB LPDDR5,最高 67 TOPS,约 102GB/s | **RGB-D 版本可做最低试验板;无 RGB-D 版本不建议交付** |

### 老 Jetson Nano 4GB

结论:不要把它写成新版最低 VLFM 的部署硬件。

原因:

1. 无 RGB-D 的模型合计 `4.28 GiB` 已经超过整机 4GB 内存。
2. 有 RGB-D 的 `2.15 GiB` 只是模型账面值,系统和运行时开销会把剩余空间吃得很快。
3. 老 Nano 是 Maxwell GPU,没有 Tensor Cores;YOLO26L TRT、SigLIP2、MobileSAM、ZoeDepth 这些模型即使勉强移植,实时性也很难看。
4. 老 Nano 软件栈停在 JetPack 4.x,CUDA/TensorRT/Python/新模型依赖会比 Orin 系列麻烦很多。

它适合做什么:

- 轻量 YOLO/分类器演示。
- 只跑底盘、传感器转发、轻量建图或低频调试节点。
- 把语义模型放到外部 GPU/NPU 盒子或远端服务器。

### Jetson Orin Nano Super 8GB

结论:可以作为**有 RGB-D 版本**的最低试验板,但要接受约束。

建议约束:

- 必须去掉 ZoeDepth,也就是 RGB-D / 深度相机要可靠。
- YOLO 用 TensorRT FP16/INT8,并在目标板上重新导出 engine。
- MobileSAM / SigLIP2 不要每帧全量跑,按策略触发或低频运行。
- 用 `tegrastats` 或 `jtop` 做 10-30 分钟稳态记录,确认没有 swap 抖动和热降频。

不建议做无 RGB-D 版本的交付板,因为 ZoeDepth 让模型合计到 `4.28 GiB`,再加运行时开销后 8GB 也会变成 "能启动但不稳" 的状态。

## 13.3 常用端侧芯片路线

| 路线 | 代表硬件 | 能承载什么 | 对 VLFM 的适配结论 |
|---|---|---|---|
| NVIDIA Jetson Orin | Orin Nano 8GB,Orin NX 8/16GB,AGX Orin 32/64GB | CUDA / TensorRT / PyTorch,可承载多模型 | **最适合整套 VLFM 端侧化** |
| NVIDIA Jetson Thor | Jetson T4000/T5000/AGX Thor | 大模型、多传感器、高并发机器人 | 能跑,但对当前最低 VLFM 明显过剩 |
| Qualcomm Snapdragon / Dragonwing | RB3 Gen 2,RB5,手机/机器人 SoC | QNN/QAIRT 跑 YOLO、分类、轻量视觉模型 | **适合替换 YOLO 服务,不适合直接搬整套 CUDA 栈** |
| Hailo | Hailo-8 26 TOPS,Hailo-8L 13 TOPS | INT8 检测模型,多路视频 YOLO | **YOLO-only 加速器**,需要主控 CPU/SoC 配合 |
| Rockchip RK3588 | Orange Pi 5,Radxa Rock 5B,Khadas 等 | RKNN 跑轻量 YOLO/分类/分割 | 低成本 YOLO/视觉节点,不适合整套 VLFM |
| Intel x86 + OpenVINO | Core Ultra / NUC / 工控机 | CPU/iGPU/NPU 跑 YOLO,也可做部分 ONNX 模型 | 适合开发、CPU fallback、小型工控机;全栈实时仍建议加独显 |
| Google Coral Edge TPU | Coral USB/M.2 | TFLite INT8 小模型 | 太窄,不建议作为本项目主路线 |

一个好用的记法:

- **想整套 VLFM 都在端侧跑:** 优先 NVIDIA Jetson Orin/AGX。
- **只想把 YOLO 从 GPU 上抠出去:** 可以看 Qualcomm QNN、Hailo、RKNN、OpenVINO。
- **想最低功耗:** 主控跑建图/控制,YOLO 上 NPU/ASIC,语义重模型降频或外置。

## 13.4 YOLO 专用适配路线

VLFM 现在的架构对 YOLO 很友好:检测器是独立服务,下游只认这个 JSON:

```json
{"boxes": [[x1, y1, x2, y2]], "logits": [0.91], "phrases": ["chair"]}
```

所以硬件适配时,优先保持 `/yolov7` 服务契约不变,只换后端推理 runtime。

### NVIDIA TensorRT

适合:

- Jetson Orin / AGX Orin / 桌面 NVIDIA GPU。
- 当前你已经测过的 `YOLO26L TRT` 路线。

注意:

- TensorRT engine 要在目标设备上重新导出,不要拿 H20/4090 上的 engine 直接放到 Jetson。
- Jetson 上优先 FP16;INT8 要做校准和精度回归。

### Qualcomm QNN / QAIRT

适合:

- Snapdragon 手机、Qualcomm RB3/RB5/RB6 类机器人平台。
- 已经有 Qualcomm 平台供应链,或 leader 特别提 "高通 NPU" 的场景。

怎么接:

- YOLO `.pt` 导出到 QNN/ONNX Runtime QNN 路线。
- 后端跑在 Hexagon NPU / Adreno GPU / Kryo CPU 上。
- 外层包一层 HTTP 服务,继续吐 VLFM 需要的 `boxes/logits/phrases`。

坑:

- QNN 路线不是 TensorRT,不能复用 TRT engine。
- YOLO26 的端到端/NMS-free 输出在 QNN、RKNN、Edge TPU 等格式上可能 fallback 到传统 one-to-many head,这时需要 NMS,不能照搬 TensorRT 的后处理。
- 量化会改变置信分布,要重新标定策略侧阈值。

### Hailo-8 / Hailo-8L

适合:

- "只跑 YOLO 检测,功耗要低" 的相机盒子/边缘网关。
- 多路视频检测,比如 Hailo-8 26 TOPS,典型功耗低。

怎么接:

- YOLO `.pt -> ONNX -> HAR -> HEF`。
- 用 HailoRT 跑 HEF,可把 NMS 配到 Hailo pipeline。
- 外层同样包装成 VLFM 的 YOLO 服务。

坑:

- Hailo 是检测模型加速器,不是通用 CUDA GPU。
- Hailo 官方/Ultralytics 文档重点支持 detection;分割、姿态、语义分割要单独确认。
- 编译要校准集,NMS JSON / model script / 输入尺寸 / 类别数必须和模型匹配。

### Rockchip RKNN

适合:

- RK3588 这类低成本 ARM 主板。
- 跑轻量 YOLO、摄像头接入、简单语义节点。

怎么接:

- YOLO `.pt -> ONNX -> RKNN`。
- 用 RKNN Runtime 跑在 NPU 上。
- 外层包装成同一检测服务。

坑:

- RK3588 NPU 量级通常是 6 TOPS,适合 YOLO-n/s 这类轻量模型,不要期待它吃完整 VLFM。
- 算子、量化、NMS、动态 shape 都要按 RKNN 工具链重做。

### CPU / OpenVINO

适合:

- x86 工控机、NUC、笔记本、Intel Core Ultra。
- 没有 NVIDIA GPU 时的 YOLO fallback。

怎么接:

- YOLO 导出 OpenVINO IR。
- 运行在 Intel CPU / iGPU / NPU。
- 包成同一 HTTP 服务。

坑:

- CPU 能跑不等于闭环导航实时可用。
- 用 CPU 跑 MobileSAM/ZoeDepth/SigLIP2 只能当调试兜底,不建议当产品方案。

## 13.5 CPU 在端侧应该承担什么

CPU 的合理角色:

- ROS / 传感器 I/O / 相机解码。
- 建图、坐标变换、frontier、路径规划。
- YOLO/SAM 的后处理和 HTTP glue。
- 小模型 fallback 或低频调试。

CPU 不适合承担:

- 每帧 MobileSAM。
- 每帧 ZoeDepth。
- 高频 SigLIP2 图文相似度。
- YOLO26L 这类模型的实时主推理。

一句话:CPU 可以管系统和胶水,不要让 CPU 扛完整语义栈。

## 13.6 推荐部署方案

### 方案 A: 当前无 RGB-D,追求一台机器跑完整语义栈

推荐:

- Jetson Orin NX 16GB 起步。
- 更稳:AGX Orin 32GB/64GB。

理由:

- 你的模型账面合计 `4.28 GiB`,工程上至少要 2 倍左右的统一内存余量才舒服。
- ZoeDepth 是最大头,无 RGB-D 时无法绕开。
- Orin 系列可以继续用 CUDA/TensorRT/PyTorch,迁移成本最低。

### 方案 B: 未来有 RGB-D,做最低成本端侧试验

推荐:

- Jetson Orin Nano Super 8GB 可试。
- Orin NX 8GB/16GB 更稳。

理由:

- 去掉 ZoeDepth 后模型合计 `2.15 GiB`。
- 8GB 统一内存有机会容纳模型、系统和相机缓存。
- 仍要做分时运行和低频触发,不要把所有模型当视频流每帧跑。

### 方案 C: leader 想听 "高通 / YOLO 专用芯片"

推荐表述:

> 高通 QNN、Hailo、Rockchip RKNN、OpenVINO 都可以作为 YOLO 检测服务的端侧加速后端。它们适合替换 VLFM 里的 YOLO RPC 服务,但不能直接替代 MobileSAM、SigLIP2、ZoeDepth 这一整套 CUDA/PyTorch 语义栈。若要整套模型本机运行,优先选 NVIDIA Jetson Orin;若只优化检测功耗,再考虑 Qualcomm/Hailo/RKNN。

## 13.7 端侧验收清单

硬件评估不要只看能不能 `import` 和跑单张图,建议按这个顺序验收:

1. **内存:** 启动所有目标服务后记录 `tegrastats` / `jtop` / `free -h`,稳态内存占用低于 80%,无持续 swap。
2. **冷启动:** 记录从开机到四个服务可用的时间,尤其是 TensorRT engine 加载和模型 warmup。
3. **稳态延迟:** 每个服务连续 100 次请求,记录 P50/P95/P99,区分首帧和 warm 后。
4. **热稳定:** 连续跑 10-30 分钟,记录温度、频率、功耗、是否降频。
5. **检测契约:** 每种后端都要检查 `boxes` 是否归一化、`phrases` 是否 COCO 类名、RGB/BGR 是否正确、NMS 是否重复或缺失。
6. **导航闭环:** 同一场景同一目标做 A/B,不要只看单帧检测速度。
7. **RGB-D 标定:** 若用 RGB-D 替代 ZoeDepth,必须验相机内参、外参、深度和彩色对齐,否则省下显存但目标点会漂。

---

## 13.8 可以直接放到汇报里的短版

> 目前无 RGB-D 的最低 VLFM 语义栈模型峰值合计约 4.28 GiB;未来接入 RGB-D 后去掉 ZoeDepth,合计约 2.15 GiB。老 Jetson Nano 4GB 不适合作为完整部署硬件;Jetson Orin Nano 8GB 可作为 RGB-D 版本最低试验板,无 RGB-D 版本建议从 Jetson Orin NX 16GB 起步。高通 QNN、Hailo、RKNN、OpenVINO 更适合作为 YOLO 检测服务的专用加速路线,不能直接替代 MobileSAM/SigLIP2/ZoeDepth 全栈。真正能大幅降低硬件门槛的是上 RGB-D,因为它直接移除 ZoeDepth 这项最大显存和延迟开销。
