子文档 01 · PointNav ResNet 内部结构

配套：数据链全景 README §6 / §8。本文专门展开 VLFM 用到的 WrappedPointNavResNetPolicy 那个"出 (1,1) long 动作"的小网络，从输入张量一路捋到 softmax 之前。

一句话：这是一个 depth-only + (ρ,θ) 的 ResNet-18 + 2 层 LSTM(512) 离散策略， ckpt 文件是 data/pointnav_weights.pth，来自 Habitat DD-PPO 训练；VLFM 自己没有重训它。

2. 三个加载分支（容易踩坑）

文件 vlfm/policy/utils/pointnav_policy.py 里有 三套 实现，按环境自动挑：

条件实际类backbone 来源动作分布VLFM 是否常用
装了 habitat-lab，habitat.__version__ == "0.1.5" habitat PointNavResNetPolicy 包一层，但 net 被覆盖成 nh_pointnav_policy.PointNavResNetNet(discrete_actions=True) vlfm 自带 resnet18（GroupNorm 版） Categorical(4) SemExp 复现走这条
装了较新 habitat（默认配置） habitat 原版 PointNavResNetPolicy.from_config(ckpt["config"]) habitat-baselines 自带 resnet 由 ckpt config 决定，HM3D 权重是 Categorical 4 类离散 ✅ HM3D 默认
没装 habitat-lab（reality / 真机） vlfm 自带 nh_pointnav_policy.PointNavResNetPolicy vlfm 自带 resnet18 GaussianNet (μ, σ) 连续 2 维（线速 + 角速） 真机 / Spot 控制走这条
这就是为什么 WrappedPointNavResNetPolicy.__init__ 里要写 discrete_actions = not hasattr(self.policy.action_distribution, "mu_maybe_std")： 连续策略才有 mu_maybe_std，离散策略没有；据此决定 pointnav_prev_actions 是 (1,1) long 还是 (1,2) float32。HM3D 评测下永远是离散。

3. 离散策略 forward 全展开（HM3D 默认）

下面以 habitat 原版 PointNavResNetPolicy（discrete）为准。每个张量的 shape 都按 VLFM 实际 batch=1、224×224 标了。

3.1 输入

observations = {
  "depth":                       (1, 224, 224, 1) float32 ∈ [0,1]   # 已被 image_resize area
  "pointgoal_with_gps_compass":  (1, 2)            float32           # (ρ 米, θ 弧度)
}
rnn_hidden_states:   (1, num_recurrent_layers=4, 512)  float32   # 2 层 LSTM × (h, c) = 4
prev_actions:        (1, 1)                            long       # 上一步动作 id ∈ {0,1,2,3}
masks:               (1, 1)                            bool       # episode 起始 0，其余 1

3.2 视觉支路

蓝框 = 普通层；黄框 = ResNet stage (BasicBlock with GroupNorm)；绿框 = 输出。 关键尺寸链：224 → avgpool ⇒ 112 → conv7s2 ⇒ 56 → maxpool ⇒ 28 → /2/2/2 ⇒ 4 → conv3 128ch ⇒ Flatten 128·4·4=2048 → Linear 512。
🔬 跟 torchvision ResNet-18 的关键差别
  • 用 nn.GroupNorm 替代 BatchNorm（DD-PPO 经典做法，rollout 里 batch 太小用 BN 不稳）。
  • base_planes 只有 32（torchvision 是 64），所以最终通道是 256 而不是 512，参数量减半。
  • 开头加了一道 avg_pool2d(k=2)，相当于先把 224 直接降到 112，再走 conv7s2，所以最终 spatial 是 4×4 而不是 7×7。
  • 没有全局平均池化，而是 1×1 conv 256→128 + flatten，明显是为了让 LSTM 看到一点空间结构。

3.3 goal + 上一步动作 + LSTM

关键细节：

  • goal_observations 的三元化：原本 (ρ,θ) 两维，被改写成 (ρ, cos(-θ), sin(-θ))，这样 LSTM 看到的是 一个角度连续可微的 3 维向量，避免 θ 从 +π 跳到 -π 时梯度炸。
  • prev_actions 的 mask 技巧：episode 开始时 masks=0， torch.where(masks, prev+1, start_token=0) 直接把 prev 替换成 dummy id 0； Embedding 表是 (4+1, 32)，多出来那一行就是这个 "start" 符号。
  • LSTMStateEncoder.forward 里有 masks.view(1,-1,1) reset hidden：episode 起始/done 后那一步会把 (h, c) 强制清零， 这就是 WrappedPointNavResNetPolicy._pointnav 里换目标时要传 masks=0 的根本原因。
  • hidden state 张量打包：habitat 把 (h, c) 沿 layer 维 torch.cat 成一个 (1, 2·num_layers, 512) 的张量在外面带着走， WrappedPointNavResNetPolicy 里就是 (1, 4, 512)。

3.4 动作头

模式head输出动作含义
HM3D 默认（discrete） Linear(512, 4) → Categorical id ∈ {0:STOP, 1:MOVE_FORWARD, 2:TURN_LEFT, 3:TURN_RIGHT} habitat turn_angle=30°，forward_step=0.25 m
真机/Spot（continuous） GaussianNet(512, 2)：μ=tanh(Linear), σ=clamp+exp (1, 2) float ∈ [-1, 1] (v_lin, v_ang)，由 reality_policy 再映射成米/秒、弧度/秒
注意 WrappedPointNavResNetPolicy.act(deterministic=True)：离散时取 argmax（实际走的是 distribution.mode）， 连续时取 μ。所以 VLFM 评测里 PointNav 是确定性的，整条管线确定性可复现。

4. (h, c) 的生命周期 / 何时 reset

WrappedPointNavResNetPolicy.__init__:
    pointnav_test_recurrent_hidden_states = zeros(1, 4, 512)
    pointnav_prev_actions                 = zeros(1, 1, long)

每帧 BaseObjectNavPolicy._pointnav(goal, stop):
    if 第 0 步:                masks = 0
    elif |goal - last_goal|>0.1m: self._pointnav_policy.reset(); masks = 0   # ← 关键
    else:                       masks = 1
    action = self._pointnav_policy.act(obs, masks)
    self._pointnav_policy.pointnav_prev_actions          ← action.clone()
    self._pointnav_policy.pointnav_test_recurrent_hidden ← rnn_hidden_states
这就是为什么 VLFM 在 navigate → explore 切换、或者目标点云中心跳了一下时，PointNav 经常会出现"突然走神又重新瞄准"的现象—— hidden 被清零，LSTM 失忆，要靠 ResNet 这一帧的特征独立判断。这点 ObjectPointCloudMap.get_best_object 的 0.1m 滞回阈值就是专门为它做的：能不抖就别抖。

5. 训练背景（不影响推理但有助于直觉）

  • 这套权重是 DD-PPO (Wijmans et al., 2020) 在 Gibson + MP3D PointGoal 上跑出来的——纯 depth 输入，奖励是 "靠近 goal 距离差 + STOP 奖励"。
  • VLFM 直接 freeze 当 low-level controller 用，自己只决定 goal 是哪儿（YOLO 找到的物体 / 选中的 frontier）， 什么时候发 STOP（只有 navigate 模式下、距 goal <0.9m 才允许，由策略外层判断，不走网络）。
  • 对应论文：VLFM 把 PointNav 当 "skill"，整个 ObjectNav 决策上层完全 zero-shot，只这一处用了预训练 RL 权重。
