子文档 02 · reveal_fog_of_war 算法细节

配套：数据链全景 README §4.2 第 9 步。本文专门拆 frontier_exploration.utils.fog_of_war.reveal_fog_of_war， 它是 ObstacleMap.update_map 里把"扇形 FOV 锥被障碍物切掉之后还可见的那一块"标到 explored_area 的核心算子。

一句话：它不是普通的"光栅 ray casting 一根一根射线"，而是 "画扇形 → 用 obstacle 把扇形撕碎 → 取含 agent 的那块多边形"，用 OpenCV 的 ellipse / polylines / findContours 几步就完事。 速度优势：500 次平均 ~几 ms 量级（脚本末尾自带 benchmark），比逐根射线快一两个数量级。

1. 入参 / 出参

def reveal_fog_of_war(
    top_down_map:              np.ndarray,  # (H, W) uint8, 1=可航行(free), 0=障碍/未知
    current_fog_of_war_mask:   np.ndarray,  # (H, W) uint8, 历史 fog（VLFM 这里每帧传 zeros，再在外面 OR 上去）
    current_point:             np.ndarray,  # (2,) int, agent 像素 (row, col)
    current_angle:             float,       # 弧度
    fov: float = 90,                        # 角度!（注意是度，不是弧度）
    max_line_len: float = 100,              # 像素
    enable_debug_visualization: bool = False,
) -> np.ndarray                            # (H, W) uint8, 0/1，新可见区
VLFM 里实际怎么传值
top_down_mapself._navigable_map.astype(uint8)（True=可走）
current_fog_of_war_masknp.zeros_like(self._map, uint8)（每帧重算，再 OR 到 self.explored_area）
current_pointagent_pixel_location[::-1]（注意 [::-1]，从 (x,y) 翻成 (row,col)）
current_angle-extract_yaw(tf_camera_to_episodic)（取负，因为像素 y 朝下）
fovnp.rad2deg(topdown_fov) = 79°（HM3D）
max_line_lenmax_depth * pixels_per_meter = 5·20 = 100 px

2. 算法 5 步走（配 SVG 图）

2.1 第 ① 步：把扇形画成一个 mask

curr_pt_cv2 = current_point[::-1].astype(int)          # 翻成 (col, row) 给 cv2 用
angle_cv2   = rad2deg(wrap_heading(-current_angle + pi/2))  # 转成 cv2 ellipse 期望的"x 轴正方向起算"
cone_mask = cv2.ellipse(
    zeros_like(top_down_map),
    curr_pt_cv2,
    (int(max_line_len), int(max_line_len)),   # 半径 a=b ⇒ 圆扇
    0,
    angle_cv2 - fov/2,
    angle_cv2 + fov/2,
    color=1,
    thickness=-1,                              # 填充
)

关键坐标变换：

  • numpy 跟 cv2 的 (row,col) vs (x,y) 反着：所以 current_point[::-1]。
  • 角度反向 + π/2 校准：episodic 坐标系 yaw 是"逆时针，x 向右"；cv2 ellipse 的 startAngle 是 "顺时针，x 轴正方向起算"；-yaw + π/2 就是把它转过来。
  • wrap_heading：把角度归到 (-π, π]，避免 ellipse 的 start/end 跨 360° 出 bug。

2.2 第 ② 步：扇形 ∩ 障碍

obstacles_in_cone = cv2.bitwise_and(cone_mask, 1 - top_down_map)   # 1=扇内且不可走
obstacle_contours, _ = cv2.findContours(obstacles_in_cone, RETR_TREE, CHAIN_APPROX_SIMPLE)

这里每一个 contour 都是"扇形内会挡住视线的一坨障碍物轮廓"。如果一个都没有 → 直接返回原 fog（说明前方畅通）。

2.3 第 ③ 步：每个障碍取最外两条切线（关键创新）

def get_two_farthest_points(source, cnt, agent_yaw):
    pts = cnt.reshape(-1, 2) - source                 # (M, 2) 相对 agent
    R = [[cos(-agent_yaw), -sin(-agent_yaw)],
         [sin(-agent_yaw),  cos(-agent_yaw)]]
    pts = pts @ R                                     # 把 agent 朝向旋到 x 轴
    angles = arctan2(pts[:,1], pts[:,0])              # (-pi, pi]
    return cnt[argmin(angles)], cnt[argmax(angles)]   # 最左最右两条切线
fallback：只有 cv2.isContourConvex(cnt) == True 时才用这一招省事； 非凸轮廓（典型：墙转角的 L 型）直接 points.append(cnt)，把整个轮廓的所有点都丢进去 ——后面靠 polylines 切碎以后再让 findContours 把"被障碍挡住的扇形碎片"合并。

2.4 第 ④ 步：用切线把扇形撕碎

visible_cone_mask = cv2.bitwise_and(cone_mask, top_down_map)   # 扇内且可走 = candidate
line_points = vectorize_get_line_points(curr_pt_cv2, points, max_line_len * 1.05)
# 每个点对 (start_point, end_point) 形成一条线段：从该 obstacle 顶点出发，沿 agent→该点 的方向延长 105%
cv2.polylines(visible_cone_mask, line_points, isClosed=False, color=0, thickness=2)

为什么 thickness=2 而不是 1？因为 findContours 要靠这条线把多边形断开，1px 容易因为浮点抖动留个缝，导致两个本该独立的区域被串成同一个，最后 pointPolygonTest 取错。

2.5 第 ⑤ 步：取含 agent 的那块多边形

final_contours, _ = cv2.findContours(visible_cone_mask, RETR_TREE, CHAIN_APPROX_SIMPLE)
visible_area, min_dist = None, inf
for cnt in final_contours:
    dist = abs(cv2.pointPolygonTest(cnt, agent_pixel, measureDist=True))
    if dist < min_dist:
        min_dist, visible_area = dist, cnt

if min_dist > 3:
    return current_fog_of_war_mask   # 离最近 contour 都超 3 px，认为出错了，跳过本帧
new_fog = cv2.drawContours(current_fog_of_war_mask, [visible_area], 0, 1, -1)
return new_fog
  • pointPolygonTest(..., True) 返回带符号距离：正=点在内，负=点在外，0=点在边上。这里取 abs 后比大小， 因为 agent 像素有时候因为四舍五入会刚好落到 1px 厚的边线上变负。
  • min_dist > 3 直接放弃这一帧：典型场景是 agent 站到了 obstacle 边缘里，所有 contour 都离它老远， 这时候宁可不更新，也不要把错误的可见区染进 explored_area。

3. ObstacleMap 这一层的额外加工（VLFM 自己加的）

vlfm/mapping/obstacle_map.py · update_map 拿到 reveal_fog_of_war 返回值之后还有 3 步：

new_explored_area = cv2.dilate(new_explored_area, np.ones((3,3), uint8), iterations=1)   # ① 放大一点
self.explored_area[new_explored_area > 0] = 1                                            # ② OR 累计
self.explored_area[self._navigable_map == 0] = 0                                         # ③ 障碍区强制清掉

contours, _ = cv2.findContours(self.explored_area, RETR_EXTERNAL, CHAIN_APPROX_SIMPLE)
if len(contours) > 1:
    # 取含 agent 的连通块（如果 agent 不在任何块内就取离它最近的那块）
    ...
    self.explored_area = new_area.astype(bool)
步骤目的
① dilate 3×3视觉上让相邻帧的可见区"咬"起来，避免几帧之间出现 1 px 缝（看起来像没探索过）
② OR 累计fog-of-war 是单调递增的：见过的地方永远算见过
③ 障碍清零第 ① 步 dilate 会侵犯到 obstacle 里，这里把 dilation 溢出的部分扣回去
含 agent 连通块过滤避免"在 A 房间穿过门 → 门外被错误标成已探索"的孤岛，比如 reveal_fog_of_war 在房屋之间打穿一条缝

4. 关键参数对最终 frontier 的影响

参数默认（HM3D）调大会怎样调小会怎样
topdown_fov79°（=camera_fov）探索区椭圆张得更开，frontier 退得更远；但容易把侧后方未观测区误判成已见 更保守，每步可见区窄一条带，frontier 集中在正前方
max_depth5 m ⇒ max_line_len=100 px看得更远，远距离 frontier 提前消失；但 BLIP2 锥也跟着延长，会有空想价值 探索区只在 1-2 m 内增长，frontier 紧贴 agent，循环 frontier 更频繁
射线 thickness2 px切得更狠，碎片更多，可能漏掉细长走廊1 px 容易粘连，对面房间会被误判可见
step 末尾 dilate kernel3×3边界更平滑，但 explored_area 会"流"过 obstacle边界破碎，frontier 会探测到伪边界

5. 跟 frontier 检测的衔接

_get_frontiers 再把这张已探索图 dilate 一次（5×5），交给 frontier_exploration.frontier_detection.detect_frontier_waypoints(navigable, explored, area_thresh)：

explored_area_dilated = cv2.dilate(explored_area, 5x5)
frontiers_px = detect_frontier_waypoints(navigable_map, explored_area_dilated, area_thresh_in_pixels)
frontiers    = self._px_to_xy(frontiers_px)

这一步等价于：

  1. 找 navigable ∩ ¬explored 的轮廓 → "可走但没见过的"边界；
  2. 面积小于 area_thresh（默认 1.5 m² = 600 px²）的过滤掉，避免噪点；
  3. 每个轮廓上取若干"代表点"（中点 + 弯折点）作为 waypoint。
因此，fog_of_war 的精度直接决定 frontier 的数量和位置： 探得太保守 → 处处是 frontier，策略乱跳； 探得太激进 → frontier 集中到很远的边角，PointNav 走到一半发现根本不可达，被 acyclic enforcer 屏蔽。 调过 HM3D 默认参数后，每步典型 frontier 数 0-4 个。
