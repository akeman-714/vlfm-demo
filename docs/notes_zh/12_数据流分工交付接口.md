# 12 · 数据流 · 分工交付接口（真机 / Spot 版）

> 这份文档**不讲算法、不写公式**。只干一件事:把真机(Boston Dynamics Spot)上 VLFM 的数据流拆成几个**交付接口**,每个接口一句话说清"数据长啥样",这样 producer 和 consumer 各干各的、互不依赖对方的实现。
>
> **全文精神(一个例子)**:障碍图 = **一张 2D 的 0/1 栅格**。说清这一句——
>
> - 做感知的同事:负责"用机身深度(或 LIDAR / SLAM)产出这张 0/1 栅格",别的不用管。
> - 做语义的我:负责"往同一坐标系的图上投语义相似度",**不碰深度,也不管点云怎么来的**。
>
> **仿真↔真机只差两处**:最上游的「传感器」和最下游的「动作」。中间三张 BEV 图 + 三条缝**完全一样**——这正是分缝切对了的证据。本文写真机这两端;要全字段/dtype 看 [08_Reality真机数据格式端到端变化链.md](08_Reality真机数据格式端到端变化链.md),仿真版看 [05_数据形态速查表.md](05_数据形态速查表.md)。

## 2. 核心交付接口清单

每个接口只回答三件事:**长啥样 / 谁产 / 谁用**。

### ① 障碍图 —— 头号分工缝 ⭐

- **长啥样**:一张 2D 俯视 **0/1 栅格**。`1000×1000`,每格 5 cm,覆盖 50 m × 50 m,原点 = 任务起点(episodic)。`1` = 有障碍,`0` = 空。真机按高度 `0.1–1.5 m` 过滤出障碍。
- **谁产**:感知 / 建图组。真机用 **5 路机身深度相机**(前 10 步用全部 5 路做 360° 建图,之后只留前左 / 前右两路省算力)。换 LIDAR / SLAM costmap 也行,只要吐出这张栅格。
- **谁用**:运动组拿它规划 / 避障;语义组拿它当"画布坐标系",把语义画在同样的格子上。
- **契约**:consumer 只认"坐标系 + 0/1"。producer 用机身深度还是 LIDAR、怎么算的,consumer 一概不管。

### ② 可行图 / 已探索图 / frontier —— 障碍图的同坐标系衍生

- **长啥样**:都和障碍图**同尺寸、同坐标系**。可行图 = 0/1(1 可走);已探索图 = 0/1(1 已知);frontier = 一串 `(x,y)` 米,标"已知与未知的交界、值得去探的方向"。
- **谁产**:感知 / 建图组(出障碍图时顺手出)。
- **谁用**:决策组在 frontier 里挑"最该去的那个";运动组拿可行图避障。
- **契约**:坐标系和障碍图一致即可对齐。

### ③ 价值图 —— 语义组的产出,叠在同坐标系上 ⭐

- **长啥样**:**又一张 2D 俯视栅格,和障碍图同坐标系、同尺寸**,但每格存的不是 0/1,而是一个 `float` 语义分("这片地方像不像有目标")。
- **谁产**:语义组。**只需要 臂上眼 RGB + 位姿 + 一个相似度模型**,把"前方画面和目标有多像"投到对应格子。**不碰机身深度、不参与障碍图的产生**——只借它的坐标系。
- **谁用**:决策组拿它给 frontier 打分,选下一个探索方向。
- **契约**:坐标系和障碍图对齐就行。语义模型可换(BLIP2 / SigLIP / 任意),"每格一个语义分"这件事不变。

### ④ 物体点云 / 目标点 —— 检测组的产出

- **长啥样**:目标的一堆 3D 点 `(x,y,z)` 米;对外只交一个**最近可信点 `(x,y)`**——"目标大概在这"。
- **谁产**:检测组(臂上眼 RGB 过检测框 + 分割,深度用 ZoeDepth 估,再反投影)。
- **谁用**:决策组判断"找到没",运动组当终点。
- **契约**:交一个米制 `(x,y)` 即可,下游不关心怎么检出来的、用哪个检测器。

### ⑤ 动作 —— 运动组的输入 / 输出 ⭐（真机和仿真差最大的一处）

- **长啥样**:
  - **输入** = 一个米制 `(x,y)` 目标(来自 frontier 或物体点)。
  - **输出** = 一个 **`(1,2)` 连续浮点 ∈ [-1,1]**,拆成 `{"angular", "linear"}`,再缩放成**单步相对位移**(`≤ 0.25 m` 平移 / `≤ 30°` 转角),喂给 `spot.set_base_position()` 走底盘。
- **谁产 / 用**:运动组自包含。上游只管给一个点,它负责走过去并在够近(`0.9 m`)时停。
- **契约**:给点 → 还动作。中间用 PointNav 网络还是别的,上游不管。
- ⚠️ **这是真机 vs 仿真唯一真正分叉的接口**:仿真出**离散动作** `long {0,1,2,3}`(停 / 前 / 左转 / 右转)直接喂仿真器;真机出**连续 `[-1,1]`** 喂 Spot 底盘。下一节专门拆这条缝。

## 3. 动作接口:共享 `_pointnav()` 的两侧分叉

到目标点之前的算路是**两端共享**的(`base_objectnav_policy.py:_pointnav`:把目标转成 `(ρ,θ)` + 拼前置深度,喂同一个 PointNav 网络)。只有**网络头和动作落地**分叉:

```
        共享: _pointnav()  —— 目标(x,y) → (ρ,θ) + nav_depth → PointNav 网络
                        │
        ┌───────────────┴────────────────┐
   仿真(Habitat)                     真机(Spot)  ← 本文路径
   Discrete(4) 头                    GaussianNet(2)+tanh 头
   输出 long {0,1,2,3}               输出 (1,2) float ∈ [-1,1]
        │                                │
   HabitatMixin.act                 RealityMixin.act
   → 离散动作                        → {"angular","linear"}
        │                                │
   仿真器 step                       _compute_displacements
   (停/前/左/右)                     clip×max_dist → 米 / 弧度
                                        │
                                    spot.set_base_position()  相对位移走底盘
```

> 交付含义:**上游(决策/语义)只交一个目标 `(x,y)`,完全不用知道下游是仿真离散头还是真机连续头**。换平台 = 换这一截网络头 + act,缝以上一行不改。

## 4. 分工怎么切:三条缝

| 缝 | 上面是谁 | 下面是谁 | 只约定这个接口 |
|---|---|---|---|
| **A** ⭐ | 语义 / 决策 | 感知 / 建图 | **BEV 障碍图 + frontier**(0/1 栅格 + (x,y) 列表) |
| B | 决策 | 检测 | **目标点 (x,y)** |
| C ⭐ | 决策 | 运动 | **目标点 (x,y)** →(运动组内部出连续动作)→ `set_base_position` |

**缝 A 是核心(就是你说的那个)**:下面随便机身深度 / LIDAR / SLAM,只要产出"2D 0/1 栅格 + frontier";上面只在这张栅格坐标系里画语义、挑方向,**一行传感器代码都不用碰**。
**缝 C 是真机和仿真唯一真正不同的地方**:接口仍是"一个目标点",只是运动组内部把它落成 Spot 的连续底盘指令。

## 5. 一句话交付卡（贴工位 / 交付时直接引用 · 真机）

| 数据 | 一句话长啥样 | 坐标系 | 交给谁 |
|---|---|---|---|
| 障碍图 | 2D 0/1 俯视栅格,5 cm/格(机身深度产出) | episodic BEV | 运动 + 语义 |
| 可行 / 已探索图 | 同坐标系,0/1 | 同上 | 运动 / 决策 |
| frontier | 一串 (x,y) 米 | 同上 | 决策 |
| 价值图 | 同坐标系,每格一个语义分 float(臂上眼 RGB 产出) | 同上 | 决策 |
| 目标点 | 一个 (x,y) 米 | episodic | 决策 / 运动 |
| 动作 | (1,2) float ∈ [-1,1] → ≤0.25m / 30° 相对位移 | 机器人本体 | Spot 底盘 |

## 6. 真机特有的两个提醒（交接时容易踩）

1. **深度有两个来源,别混**:障碍图 / 导航用的是**机身深度相机的真实深度**;臂上眼**没有对齐深度**,价值图 / 检测要用深度时是 **ZoeDepth 单目临场估**的(只在那帧检测到物体才估)。两路别接错。
2. **开局先扫臂环视**:正式导航前,机械臂会把臂上眼依次摆到 `[-90°…90°]` 扫一圈先建一圈价值图(`INITIAL_ARM_YAWS`),这期间底盘不动、动作字段是 `arm_yaw` 而非 `angular/linear`。下游解析动作时要区分"初始化阶段 vs 导航阶段"。
