子文档 03 · ITMPolicyV3 双通道 exploration_thresh 逻辑

配套：数据链全景 README §3 / §4.3 / §4.7。 本文专门拆 V3 比 V2 多出来的"两个 prompt + 一个阈值"那套机制——它在 HM3D 默认 yaml 里被关闭（exploration_thresh=0）， 但在论文消融、SemExp 复现、reality 配置里都会被打开，搞清楚它对调 hyper-param 很关键。

一句话：V3 用 两个 prompt（"目标 prompt" + "探索 prompt"）让 BLIP2-ITM 给每帧打两个余弦， 在 ValueMap 里以 双通道 累计；选 frontier 时检查"目标通道的全局最大值是否过 exploration_thresh"， 过了就只看目标通道，没过就只看探索通道。这是个二态切换，不是混合，也不是加权。

1. 类关系 / 默认配置一览

BaseITMPolicy (itm_policy.py)
├── ITMPolicy        (V1, FrontierMap，每个 frontier 缓存一次 cosine)
├── ITMPolicyV2      (V2, 单通道 ValueMap, 每步全图 cosine)
│    └── ITMPolicyV3 (V3, 双通道 ValueMap + exploration_thresh)
│         └── HabitatITMPolicyV3 = HabitatMixin + ITMPolicyV3   (vlfm/policy/habitat_policies.py)

启用方式（SemExp 评测脚本 vlfm/semexp_env/eval.py）：

exp_thresh = float(os.environ.get("EXPLORATION_THRESH", 0.0))
if exp_thresh > 0.0:
    policy_cls = SemExpITMPolicyV3
    policy_kwargs["exploration_thresh"] = exp_thresh
    policy_kwargs["text_prompt"] = (
        "Seems like there is a target_object ahead."
        "|There is a lot of area to explore ahead."
    )
else:
    policy_cls = SemExpITMPolicyV2
HM3D 默认就是 V2。V3 的两个 prompt 用 "|"（PROMPT_SEPARATOR）拼接， ValueMap.__init__ 用 len(text_prompt.split(PROMPT_SEPARATOR)) 决定通道数—— 所以"是不是 V3"完全由 prompt 里有没有 | 决定，类名只是个壳。

2. 两 prompt 在数据链里的走向

读图重点：

  • 两个 prompt 共用同一张 RGB，BLIP2-ITM 被调两次，得到 cosines = (cosine₁, cosine₂)。
  • 它们一起调 ValueMap.update_map(np.array(cosines), depth, tf, ...)， 在每个通道里都按 同一个锥形置信度 + 同一张 self._map（confidence） 加权融合， 只是 self._value_map[..., 0] 累计 cosine₁，self._value_map[..., 1] 累计 cosine₂。
  • 置信度只有一份——这是 V3 在实现上最关键的细节：两个通道共享同一个 self._map（confidence）， 因为它们对应的可见区/遮挡情况完全一样，权重也就一样。

3. _reduce_values：唯一一处真正用到 exploration_thresh 的地方

# vlfm/policy/itm_policy.py · ITMPolicyV3
def _sort_frontiers_by_value(self, observations, frontiers):
    sorted_frontiers, sorted_values = self._value_map.sort_waypoints(
        frontiers, radius=0.5, reduce_fn=self._reduce_values
    )
    return sorted_frontiers, sorted_values

def _reduce_values(self, values: List[Tuple[float, float]]) -> List[float]:
    """values 是每个 frontier 的 (target_value, exploration_value)。"""
    target_values    = [v[0] for v in values]
    max_target_value = max(target_values)
    if max_target_value < self._exploration_thresh:
        # 所有 frontier 的"目标价值"都太低 → 当前一帧也没看出目标在哪
        # 退化为"找还没探索完的大区域"
        return [v[1] for v in values]            # exploration_value
    else:
        # 至少一个 frontier 的目标价值 >= 阈值 → 信任 BLIP2 关于"目标在哪"的判断
        return [v[0] for v in values]            # target_value

3.1 决策树

这是个 硬切换，不是混合。所以 exploration_thresh 调过头会出现： 本帧 target_value max 是 0.0008，恰好越过 0.0007 阈值，整个排序立刻翻转—— 上一帧选的 frontier 突然从 "高 explore" 变成 "低 target"，PointNav 目标骤变，hidden 被 reset（详见 PointNav 子文档 §4）。所以阈值要尽量挑在 BLIP2 cosine 的双峰之间。

3.2 exploration_thresh = 0 时

max_target_value < 0   →  False           # 因为 cosine ≥ 0
所以永远走 if-else 的 else 分支          → 按 target_value 排

结论：V3 + thresh=0 退化为 V2，但 BLIP2 多算一次（白费算力）；除非你要做消融，否则别这么设。

3.3 exploration_thresh = 1.0 时

BLIP2 cosine 极少 ≥ 1，所以几乎永远走 if 分支——退化为 "frontier 永远按 explore prompt 排序"。 这跟 frontier-based exploration（FBE）的行为很接近，只是 explore 通道是 BLIP2 给的"是不是宽阔区域"评分， 而不是纯几何距离/面积。

4. 可视化（visualize_value_map）

V3 改写了基类的 _vis_reduce_fn：

def visualize_value_map(arr):                  # arr: (1000,1000,2)
    first_channel = arr[:, :, 0]
    max_values    = np.max(arr, axis=2)
    mask          = first_channel > exploration_thresh
    return np.where(mask, first_channel, max_values)

语义：

  • 对每个像素单独判断"目标通道是否过阈值"——过了显示目标通道，没过显示两通道最大值。
  • 注意这是像素级判断，但 _reduce_values 是frontier-集合-级判断（全局 max），两者口径不同， 所以你看可视化"红的地方多"不等于实际排序用的就是目标通道。

5. ValueMap 内部的双通道融合（再确认一遍）

看 vlfm/mapping/value_map.py · _fuse_new_data：

# 默认 use_max_confidence=False 时：
weight_1 = self._map / (self._map + new_map)            # (H,W)
weight_2 = new_map   / (self._map + new_map)            # (H,W)
self._value_map = self._value_map * weight_1[..., None] + values * weight_2[..., None]
                                                        # ↑ weight 广播到所有通道
self._map       = self._map       * weight_1            + new_map * weight_2

关键观察：

  1. 权重张量 weight_1/weight_2 是 (H,W)，values 是 (C,) 标量； * weight_2[..., None] 把权重广播到所有通道，所以 两通道用完全相同的权重做加权平均。
  2. 新值 values = (cosine₁, cosine₂)在锥形可见区内是常数标量（不随像素变）——这是 BLIP2 给整张图的一个分数， 它的"空间结构"完全来自new_map这张置信度锥形 mask，而不是这两个值本身。
  3. sort_waypoints(radius=0.5) 在每个 frontier 周围 0.5 m 半径内取的是中位数（pixel_value_within_radius）， 所以对噪声很 robust，但也意味着阈值要按"中位数"标准来调，不能按 BLIP2 单次输出的均值/max。

6. 怎么挑 exploration_thresh

  1. 先开 V2 默认跑几条 episode，把 BLIP2ITMClient.cosine 的返回值打 log，画直方图。 HM3D chair 任务大约会看到 cosine 集中在 0.0003-0.0015，目标真的在视野内时 spike 到 0.002-0.005。
  2. 挑两个峰之间的"谷"——一般 exploration_thresh ≈ 0.0007 就是这种数量级（注意它在 BLIP2 ITM 头里实际是 未做 sigmoid 的 logits 量级，所以看着远小于 1）。
  3. 同时拿 SPL 当 sanity check：阈值偏低 → 频繁切回探索，SPL 涨；偏高 → 经常硬切，PointNav 晃悠，SPL 跌。
  4. 不要跨数据集复用阈值。HM3D / MP3D / Reality 各 prompt 长度/类别多样性都不同，cosine 分布会偏。

7. V3 的失败模式（trace 时常见）

现象可能原因排查点
BLIP2 双 prompt 一直爆 timeout 每帧 2 次 HTTP，VLM 服务被打爆 vlfm/vlm/server_wrapper.py 重试次数 / 看 12182 端口 QPS
ValueMap 视觉看着 OK 但 frontier 选错 "frontier 排序用的是另一个通道"——回看 §3.1 决策树 打印 max(target_values) 和 self._exploration_thresh
切换抖动，每隔几步换 frontier max_target 反复在阈值附近震荡 加个迟滞：阈值上下加 ±10% 死区，或者沿用 last_value 偏置
Reality 配置下 cosine 全是 0 prompt 拼接漏写了 "|"，value_channels=1 但 V3 还期望 tuple print(self._value_map.shape)，看是不是 (size,size,1)；assert 通道数

8. trace 一帧时的断点

  1. vlfm/policy/itm_policy.py · ITMPolicyV2._update_value_map （V3 继承自 V2 没重写）→ 打印 cosines，确认拿到的是 list[list[float, float]]，外层长度 1（HM3D 单相机），内层长度 2。
  2. vlfm/mapping/value_map.py · update_map → 在 _fuse_new_data 入口 print(values, values.shape)，确认 (2,) float32。
  3. vlfm/policy/itm_policy.py · ITMPolicyV3._reduce_values → 打印 max_target_value 跟阈值的相对关系， 以及最后返回的是哪个通道的列表。
  4. 对照 BaseITMPolicy._get_best_frontier 的 top_two_values 元组， 它会被 AcyclicEnforcer 用做"状态-动作"哈希；通道切换会导致 hash 突变，绕过 cyclic 抑制—— 所以阈值附近来回跳时容易陷入"原本应该 suppressed 的 frontier 又被允许"。

9. 跟论文的对应

VLFM 论文 (Yokoyama et al., 2024) Sec. 4.3 提到 V3 的两个 prompt 是为了 解耦"目标价值"和"探索价值"，避免单 prompt 在屋子完全没目标时也强行给出方向偏好。 论文实验里阈值是经验调的，作者代码里更普遍的写法是把 exploration_thresh 当环境变量传，方便扫参。

记住：HM3D 默认 V2，所以你直接跑下来 ValueMap 是单通道，_reduce_values 根本不会被调用。 想体验 V3 改两处：① 把 yaml 里的 policy name 换成 HabitatITMPolicyV3； ② 把 text_prompt 改成带 | 的双 prompt，并设 exploration_thresh > 0。
